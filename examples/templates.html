Putting all together

By now, we clearly realize that templates plays the central role, while FL remains in the shadows exposing the API to the template.

That is precisely the intention: give complete freedom to the look and feel of your application while FL provides the underlying basic functionality and an agreement for the client-server communication.

It is common to find modules that perform a different approach to this goal. These modules, usually completely encapsulate the dynamic HTML generation and expose an API to perform any customization of appearance. This leads to unwieldy APIs, only in order to allow flexibility.

FL's soul is giving maximum flexibility and we understand that this is achieved only by giving the application full control of the template.

Another main objective of FL, is to allow the creation of reusable templates. For a template to be reusable with different datasets, it should not contain any reference to fields or settings of a particular dataset.

To illustrate these concepts, and how they can be achieved, FL includes some sample templates.

You are encouraged to create your unique and wonderful templates, or modify an existing one to suit your needs.

CrudTable template

The following example uses CrudTable, a complex template (included with FL) that implements many of the API features.

This template is intended to be used in CRUD applications.

Additional Angujar.JS modules are needed in this template:
